# 为什么选择 GVIM

<Callout type="info" emoji="🦌">
  GVIM 起源于深度研究，但逐渐演化为一个通用的长时序 Agent
  运行时——支持技能、记忆、工具和协作调度。
</Callout>

GVIM 的诞生是因为现代 Agent 系统需要的不仅仅是一个聊天循环。一个真正有用的 Agent 必须能够进行长时序规划、将任务拆解为子任务、使用工具、操作文件、安全地运行代码，并在复杂任务中保持足够的上下文连贯性。GVIM 正是为提供这样的运行时基础而构建的。

## 从深度研究起步

GVIM 的第一个版本围绕一个具体目标设计：生成真正有深度的研究产出，而不是轻量级聊天机器人的概要总结。核心思路是让 AI 系统像研究团队一样工作：制定计划、收集来源、交叉验证发现，并输出有实际深度的结构化结果。

这个定位是有效的，但项目很快揭示了更重要的东西。团队不仅将 GVIM 用于研究——他们将其应用于数据分析、报告生成、内部自动化、运营工作流，以及其他同样需要多步骤执行的任务。

共同点是清晰的：有价值的部分不仅仅是研究工作流本身，而是其背后的运行时能力。

## 研究是第一个技能，而非整个系统

这种使用方式的转变带来了一个关键结论：深度研究应该被视为更广泛 Agent 运行时中的一项能力，而不是整个产品的定义。

因此，GVIM 从一个以单一研究模式为中心的项目演化为一个通用的长时序 Agent Harness。在这个模型中，研究仍然重要，但它成为众多技能中的一项，而非系统的固定形态。

这就是为什么 GVIM 被描述为 **Harness**，而不仅仅是框架或应用。

## 为什么 Harness 很重要

Harness 是一个带有主张的 Agent 运行时。它不只是暴露抽象接口。它打包了 Agent 在真实环境中做有用工作所需的基础设施——工具访问、技能加载、沙箱执行、记忆、子 Agent 协调和上下文管理。

这让开发者可以专注于他们试图解决的问题，而不是重复构建相同的基础设施层。

## GVIM 解决的问题与科学计算特性

GVIM 专注于解决化学与材料科学领域的科研探索问题：

**长时序与科学多步骤规划**：大多数有用的科学计算任务不能在单次推理中完成。GVIM 允许科学 Agent 在获取文献、处理分子结构、调用 RDKit 分析或 pymatgen 晶体结构时进行多轮自我纠错与长时序规划。

**上下文工程与科学数据管理**：随着任务变长，复杂的分子结构、CIF文件、文献文本会极快耗尽上下文。GVIM 通过范围隔离的科学子 Agent、文件系统作为外部缓存来主动管理科研数据上下文。

**面向科研的技能包（Skills）**：不同实验与分析需要不同的专业能力。GVIM 将 RDKit 分子分析、pymatgen 晶体结构计算、Materials Project 数据检索、Ketcher 化学画板等封装为可独立加载的技能（Skills），让 Agent 能够专注并实事求是地解决领域问题。

**安全的代码执行沙箱**：真实的化学与材料数据处理需要运行复杂的 Python 脚本（如数值模拟或生成分子图）。GVIM 提供安全的隔离沙箱，让科学 Agent 可以进行实际的文件读写和代码运算。
